# 文档编写规范

## 1. 概述

本文档定义了AI Voice Interaction App前端项目的文档编写规范，旨在确保项目文档的一致性、可读性和可维护性。所有项目文档都应遵循本规范编写。

## 2. 文档类型

### 2.1 核心文档
- **架构设计文档**：描述系统架构、组件设计、状态管理等
- **功能设计文档**：描述系统功能的设计和实现
- **API文档**：描述前后端API接口规范
- **技术栈文档**：描述技术选型和依赖管理
- **开发指南**：描述开发环境、流程和规范
- **测试文档**：描述测试策略、质量保证计划等

### 2.2 其他文档
- **README.md**：项目概述和快速入门指南
- **SUMMARY.md**：文档目录和导航
- **CHANGELOG.md**：版本变更记录
- **CONTRIBUTING.md**：贡献指南

## 3. 文档格式

### 3.1 文档格式
- 使用Markdown格式编写所有文档
- 遵循GitHub Flavored Markdown (GFM) 规范
- 使用UTF-8编码

### 3.2 标题层级
- 使用#到######表示标题层级
- 一级标题（#）用于文档标题
- 二级标题（##）用于主要章节
- 三级标题（###）用于小节
- 四级及以下标题用于更详细的内容

### 3.3 格式规范

#### 3.3.1 段落
- 段落之间空一行
- 每行长度建议不超过120个字符

#### 3.3.2 列表
- 使用-或*表示无序列表
- 使用数字.表示有序列表
- 列表项之间空一行
- 嵌套列表使用4个空格缩进

#### 3.3.3 代码块
- 使用```language表示代码块
- 为代码块指定正确的语言
- 代码块之间空一行
- 关键代码添加必要的注释

#### 3.3.4 表格
- 使用Markdown表格语法
- 表格标题居中对齐
- 表格内容根据需要左对齐或右对齐
- 表格之间空一行

#### 3.3.5 链接
- 使用[文本](URL)语法表示链接
- 内部链接使用相对路径
- 外部链接使用完整URL
- 链接文本应清晰描述链接内容

#### 3.3.6 图片
- 使用![alt文本](图片URL)语法表示图片
- 为图片添加描述性alt文本
- 图片URL使用相对路径

## 4. 文档结构

### 4.1 文档模板

每个文档应包含以下基本结构：

```markdown
# 文档标题

## 1. 概述

简要描述文档的目的和内容。

## 2. 章节1

详细描述章节1的内容。

### 2.1 小节1

详细描述小节1的内容。

## 3. 章节2

详细描述章节2的内容。

## ...

## N. 相关文档

列出与本文档相关的其他文档链接。
```

### 4.2 目录结构

- 核心文档存放在`core-docs/`目录下
- 按文档类型组织子目录
- 每个子目录对应一个文档类型
- 文档文件使用小写字母和连字符命名

## 5. 命名规范

### 5.1 文档文件名
- 使用小写字母和连字符（-）分隔单词
- 文件名应清晰描述文档内容
- 避免使用缩写和简写
- **示例**：`frontend-architecture-design.md`

### 5.2 目录名
- 使用小写字母和连字符（-）分隔单词
- 目录名应清晰描述文档类型
- **示例**：`architecture-design/`

## 6. 编写指南

### 6.1 内容要求
- 内容清晰、准确、完整
- 描述要具体，避免模糊和歧义
- 使用简洁、易懂的语言
- 避免使用行话和术语，必要时解释

### 6.2 结构要求
- 逻辑清晰，层次分明
- 章节之间过渡自然
- 重要内容突出显示
- 提供必要的示例和代码片段

### 6.3 风格要求
- 保持一致的写作风格
- 使用第三人称
- 使用主动语态
- 避免使用感叹号和问号
- 避免使用表情符号

### 6.4 代码规范
- 代码示例应符合项目的代码规范
- 代码示例应完整、可运行
- 代码示例添加必要的注释
- 代码示例使用正确的缩进和格式

## 7. 维护指南

### 7.1 文档更新
- 代码变更时同步更新相关文档
- 需求变更时更新相关文档
- 定期审查和更新文档
- 保持文档与代码的一致性

### 7.2 文档版本管理
- 使用Git进行文档版本管理
- 文档变更应提交到Git仓库
- 提交信息应清晰描述文档变更
- 定期合并文档变更到主分支

### 7.3 文档审查
- 文档编写完成后进行审查
- 审查内容包括准确性、完整性、可读性
- 审查通过后发布文档
- 定期进行文档审查和更新
