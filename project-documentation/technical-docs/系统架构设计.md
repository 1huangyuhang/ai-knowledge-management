# 系统架构设计

## 1. 项目本质定义

### 1.1 项目定位

本项目是一个**认知辅助型 AI 软件**，其核心目标不是传统的知识管理（CRUD），而是：

> **持续构建、更新、分析用户的「认知结构模型」，并基于该模型给出结构性反馈与思考建议。**

### 1.2 系统关注点

系统关注的不是：
- 你记了什么

而是：
- 你反复在想什么
- 你的思考主干在哪里
- 你的概念结构是否完整
- 你下一步“最值得思考”的方向

## 2. 架构设计原则

本系统采用**Clean Architecture**设计理念，遵循以下核心原则：

1. **Domain First**：认知模型优先于技术选型
2. **AI is a Dependency, not the Core**：AI 是外部能力，不是系统中心
3. **Replaceable by Design**：任何 AI / DB / 框架都可替换
4. **Single Responsibility**：每个模块只解决一个问题
5. **高内聚、低耦合**
6. **依赖倒置**：上层模块依赖接口，不依赖具体实现

## 3. 总体分层设计

系统采用严格的分层架构，依赖方向只能向下，禁止反向引用。

```
┌──────────────────────────────┐
│ Presentation Layer            │
│  - UI / API / CLI              │
└───────────────▲──────────────┘
                │
┌───────────────┴──────────────┐
│ Application Layer              │
│  - Use Cases                    │
│  - Workflow Orchestration       │
└───────────────▲──────────────┘
                │
┌───────────────┴──────────────┐
│ Domain Layer                   │
│  - Cognitive Model              │
│  - Business Rules               │
└───────────────▲──────────────┘
                │
┌───────────────┴──────────────┐
│ Infrastructure Layer           │
│  - Persistence                 │
│  - AI Providers                │
│  - Message / Event             │
└───────────────▲──────────────┘
                │
┌───────────────┴──────────────┐
│ AI Capability Layer             │
│  - LLM                          │
│  - Embeddings                   │
│  - Reasoning                    │
└──────────────────────────────┘
```

### 3.1 各层职责

| 层级 | 名称 | 职责 | 技术要求 |
|------|------|------|----------|
| 1 | Presentation Layer | 交互与展示 | HTTP API / CLI / UI |
| 2 | Application Layer | 用例与工作流编排 | 依赖倒置，只依赖接口 |
| 3 | Domain Layer | 认知模型与业务规则 | 纯净层，不依赖任何外部库 |
| 4 | Infrastructure Layer | 存储 / AI / 外部系统实现 | 实现接口，处理技术细节 |
| 5 | AI Capability Layer | 模型与推理能力 | 可替换的 AI 服务 |

## 4. 核心系统对象

### 4.1 核心对象关系图

```
UserCognitiveModel
├── CognitiveConcept[]
├── CognitiveRelation[]
└── EvolutionHistory

CognitiveConcept
├── semanticIdentity
├── abstractionLevel
└── confidenceScore

CognitiveRelation
├── sourceConcept
├── targetConcept
└── relationType

ThoughtFragment → CognitiveProposal → UserCognitiveModel
          ↑               ↑
          │               │
          └── AI Processing

CognitiveModel → CognitiveInsight → User Feedback
```

### 4.2 核心对象定义

#### 4.2.1 UserCognitiveModel

唯一标识一个用户的认知结构，包含：
- **Concept Graph**：概念图谱
- **Relation Set**：关系集合
- **Evolution History**：演化历史

#### 4.2.2 CognitiveConcept

表示一个抽象思维单元，属性包括：
- **semanticIdentity**：语义标识
- **abstractionLevel**：抽象级别
- **confidenceScore**：置信度评分

#### 4.2.3 CognitiveRelation

表示概念间的逻辑关系，类型包括：
- `depends_on`
- `generalizes`
- `contradicts`

#### 4.2.4 ThoughtFragment

用户输入的原始思维片段，是系统的基础输入。

#### 4.2.5 CognitiveProposal

AI 生成的认知结构建议，包含：
- 概念候选
- 关系候选
- 置信度

#### 4.2.6 CognitiveInsight

系统生成的结构性反馈，包括：
- 核心主题
- 思维盲点
- 概念空洞

## 5. 模块间交互流程

### 5.1 输入处理流程

1. **用户输入**：用户通过 HTTP API 提交文本
2. **存储原始输入**：系统将输入保存为 ThoughtFragment
3. **AI 认知解析**：AI 处理 ThoughtFragment，生成 CognitiveProposal
4. **模型更新**：Domain Service 处理 Proposal，更新 UserCognitiveModel
5. **事件触发**：触发 CognitiveModelUpdated 事件

### 5.2 反馈生成流程

1. **定期分析**：系统定期分析 UserCognitiveModel
2. **生成反馈**：生成 CognitiveInsight
3. **呈现反馈**：通过 API 或其他方式呈现给用户

### 5.3 思考建议流程

1. **模型分析**：分析用户的认知模型
2. **AI 建议生成**：基于模型生成思考建议
3. **建议呈现**：将建议呈现给用户

## 6. 技术完成态定义

当以下条件全部满足时，项目在技术层面被视为“完成”：

1. 系统可持续运行，不依赖人工干预
2. 任意 AI Provider 可被替换而不影响 Domain
3. Domain 层可脱离 AI 单独运行
4. 所有关键模块均有明确接口契约（Contract）
5. 项目文档可被另一 AI 直接接管并继续开发

## 7. 扩展性设计

### 7.1 允许的扩展方向

- 多用户支持
- 多模型并行
- 长期认知趋势分析

### 7.2 可替换组件

- LLM Provider
- Embedding Provider
- 数据库
- 存储技术

只需实现相应接口即可替换上述组件，不影响系统核心功能。

## 8. 系统边界与约束

- 不使用前端复杂框架（初期）
- 不引入微服务
- 不使用魔法式 Agent 框架
- 目标：**清晰 > 智能 > 炫技**

## 9. 项目完成态定义

项目在 90 天后被认为“完成”，如果：
- 系统能持续分析你的输入
- 能输出认知结构总结
- 能给出合理的思考建议
- 架构清晰、可讲清楚
