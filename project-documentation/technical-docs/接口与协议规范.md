# 接口与协议规范

## 1. API 接口设计

### 1.1 设计原则

- **RESTful 风格**：使用标准的 HTTP 方法和状态码
- **简洁易用**：接口设计简单直观，便于开发者使用
- **版本控制**：支持 API 版本管理，便于后续扩展
- **安全性**：包含适当的认证和授权机制
- **文档化**：使用 Swagger 或类似工具生成 API 文档

### 1.2 基础 URL

```
http://localhost:3000/api/v1
```

### 1.3 认证机制

- 初期使用 API Key 认证
- 后续可扩展为 OAuth 2.0 或 JWT

### 1.4 核心 API 接口

#### 1.4.1 思维片段管理

| 方法 | 路径 | 功能描述 | 认证要求 |
|------|------|----------|----------|

| GET | /thoughts | 获取所有思维片段 | 需要 |
| GET | /thoughts/:id | 获取特定思维片段 | 需要 |

```http
POST /api/v1/thoughts
Content-Type: application/json

```

```json
{
  "id": "123e4567-e89b-12d3-a456-426614174000",

  "message": "思维片段已保存"
```

## 2. AI 与系统交互协议

### 2.1 核心原则

- **AI 是外部依赖**：AI 被视为系统的外部服务，不是核心组件
- **结构化输出**：AI 必须输出结构化 JSON 数据，不能输出自然语言
- **Proposal 机制**：AI 只能生成建议，不能直接修改系统状态
- **可替换性**：支持不同 AI 提供商的无缝切换

### 2.2 交互流程

```
系统 → AI 请求（带 Prompt 和上下文）→ AI 处理 → 结构化输出 → 系统验证 → 使用结果
```

### 2.3 请求格式

系统向 AI 发送的请求必须包含以下字段：

- **prompt**：经过版本化管理的 Prompt 模板
- **context**：必要的上下文信息
- **format**：指定输出格式（JSON）
- **max_tokens**：最大输出 token 数
- **temperature**：生成温度参数

**请求示例**：
```json
{
  "prompt": "v1.0.0/cognitive-parsing",
  "context": {
    "thought": "这是一个思维片段示例",
    "existingConcepts": ["AI", "学习"]
  },
  "format": "json",
  "max_tokens": 1000,
  "temperature": 0.7
}
```

### 2.4 响应格式

AI 返回的响应必须是结构化 JSON，包含以下字段：

- **status**：请求状态（success/error）
- **data**：实际数据（根据 Prompt 不同而变化）
- **confidence**：置信度评分（0-1）
- **reasoning**：AI 推理过程（可选）

## 3. Proposal 机制

### 3.1 定义

Proposal 是 AI 生成的认知结构建议，包含概念候选和关系候选，需要经过 Domain Service 验证后才能生效。

### 3.2 CognitiveProposal 结构

```json
{
  "concepts": [
    {
      "semanticIdentity": "概念名称",
      "abstractionLevel": 1,
      "confidenceScore": 0.9,
      "description": "概念描述"
    }
  ],
  "relations": [
    {
      "sourceConcept": "源概念",
      "targetConcept": "目标概念",
      "relationType": "depends_on",
      "confidenceScore": 0.8
    }
  ],
  "confidence": 0.85,
  "reasoningTrace": [
    "步骤1：分析文本内容",
    "步骤2：提取关键概念",
    "步骤3：识别概念关系"
  ]
}
```

### 3.3 Proposal 处理流程

1. AI 生成 CognitiveProposal
2. 系统验证 Proposal 格式和完整性
3. Domain Service 验证 Proposal 的业务规则
4. 如果验证通过，更新 UserCognitiveModel
5. 如果验证失败，记录错误并返回

### 3.4 验证规则

- 概念名称必须唯一
- 关系必须在合法关系类型列表中
- 置信度必须高于阈值（默认 0.5）
- 概念必须有明确的语义标识

## 4. Prompt 工程规范

### 4.1 Prompt 设计原则

- **清晰明确**：明确告诉 AI 要做什么
- **结构化输出**：要求 AI 输出特定格式的 JSON
- **上下文相关**：提供必要的上下文信息
- **可测试性**：便于验证和测试
- **版本化管理**：便于追踪和更新

### 4.2 Prompt 版本管理

- Prompt 必须有明确的版本号（如 v1.0.0）
- 版本号格式：主版本.次版本.修订版本
- 重大变更增加主版本号
- 功能添加增加次版本号
-  bug 修复增加修订版本号

### 4.3 Prompt 存储结构

```
src/infrastructure/ai/prompts/
├── v1.0.0/
│   ├── cognitive-parsing.json
│   ├── insight-generation.json
│   └── suggestion-generation.json
└── v1.1.0/
    ├── cognitive-parsing.json
    └── ...
```

### 4.4 Prompt 示例

**认知解析 Prompt**：
```json
{
  "version": "v1.0.0",
  "name": "cognitive-parsing",
  "system": "你是一个认知分析师，负责从文本中提取概念和关系。",
  "user": "请分析以下文本，提取概念和它们之间的关系，并以JSON格式输出。\n\n文本：{text}\n\n现有概念：{existingConcepts}\n\n输出格式：\n{\n  \"concepts\": [{\"semanticIdentity\": \"\", \"abstractionLevel\": 1, \"confidenceScore\": 0.0}],\n  \"relations\": [{\"sourceConcept\": \"\", \"targetConcept\": \"\", \"relationType\": \"\", \"confidenceScore\": 0.0}],\n  \"confidence\": 0.0\n}",
  "examples": [
    {
      "input": "文本：AI是未来的重要技术\n现有概念：[]",
      "output": {
        "concepts": [{"semanticIdentity": "AI", "abstractionLevel": 1, "confidenceScore": 0.95}, {"semanticIdentity": "未来", "abstractionLevel": 2, "confidenceScore": 0.85}, {"semanticIdentity": "技术", "abstractionLevel": 1, "confidenceScore": 0.9}],
        "relations": [{"sourceConcept": "AI", "targetConcept": "技术", "relationType": "is_a", "confidenceScore": 0.95}, {"sourceConcept": "AI", "targetConcept": "未来", "relationType": "related_to", "confidenceScore": 0.85}],
        "confidence": 0.9
      }
    }
  ]
}
```

### 4.5 Prompt 评估与优化

- 定期评估 Prompt 效果
- 根据实际使用情况优化 Prompt
- 记录 Prompt 变更历史
- 保持 Prompt 的简洁性和有效性

## 5. 事件系统协议

### 5.1 事件类型

系统使用以下核心事件类型：

| 事件类型 | 触发时机 | 包含数据 |
|----------|----------|----------|
| ThoughtIngested | 思维片段存储成功 | thoughtId, content, timestamp |
| CognitiveModelUpdated | 认知模型更新成功 | modelId, timestamp, changes |
| InsightGenerated | 认知反馈生成成功 | insightId, modelId, timestamp |
| SuggestionsGenerated | 思考建议生成成功 | suggestionId, modelId, timestamp |
| AIProcessingFailed | AI 处理失败 | error, context, timestamp |

### 5.2 事件格式

```json
{
  "id": "123e4567-e89b-12d3-a456-426614174000",
  "type": "ThoughtIngested",
  "timestamp": "2023-06-01T12:00:00Z",
  "data": {
    "thoughtId": "789e4567-e89b-12d3-a456-426614174000",
    "content": "思维片段内容",
    "metadata": {}
  }
}
```

### 5.3 事件处理

- 事件采用异步处理方式
- 使用事件总线实现模块间通信
- 支持事件持久化，便于调试和审计
- 支持事件重试机制，提高系统可靠性

## 6. 数据传输规范

### 6.1 数据格式

- 所有 API 请求和响应使用 JSON 格式
- 日期时间使用 ISO 8601 格式
- 字符串使用 UTF-8 编码
- 错误信息使用统一格式

### 6.2 错误处理

**错误响应格式**：
```json
{
  "status": "error",
  "code": "INVALID_INPUT",
  "message": "输入格式无效",
  "details": {
    "field": "content",
    "reason": "内容不能为空"
  }
}
```

**常见错误码**：

| 错误码 | 描述 | HTTP 状态码 |
|--------|------|-------------|
| INVALID_INPUT | 输入格式无效 | 400 |
| UNAUTHORIZED | 认证失败 | 401 |
| FORBIDDEN | 没有权限 | 403 |
| NOT_FOUND | 资源不存在 | 404 |
| INTERNAL_ERROR | 内部服务器错误 | 500 |
| AI_ERROR | AI 处理错误 | 500 |

### 6.3 分页规范

列表查询 API 支持分页：

**请求参数**：
- `page`：页码，默认 1
- `limit`：每页数量，默认 10，最大 100

**响应格式**：
```json
{
  "data": [...],
  "pagination": {
    "page": 1,
    "limit": 10,
    "totalItems": 100,
    "totalPages": 10
  }
}
```

## 7. 接口版本管理

### 7.1 版本号格式

使用语义化版本号：`主版本.次版本.修订版本`

### 7.2 版本升级策略

- **主版本升级**：不兼容的 API 变更
- **次版本升级**：向后兼容的功能添加
- **修订版本升级**：向后兼容的问题修复

### 7.3 版本废弃策略

- 每个版本至少支持 6 个月
- 废弃前 3 个月发布通知
- 废弃后保留 3 个月的兼容层
