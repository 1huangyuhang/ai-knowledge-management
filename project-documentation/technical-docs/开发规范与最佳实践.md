# 开发规范与最佳实践

## 1. 代码规范

### 1.1 命名约定

#### 1.1.1 文件名
- 使用 PascalCase 命名类文件（如 `UserCognitiveModel.ts`）
- 使用 kebab-case 命名其他文件（如 `cognitive-model-service.ts`）
- 接口文件使用 `.ts` 扩展名，不使用 `.d.ts` 除非是外部类型定义

#### 1.1.2 变量与函数
- 使用 camelCase 命名变量和函数（如 `cognitiveModel`、`updateModel()`）
- 使用 PascalCase 命名类和接口（如 `CognitiveConcept`、`ThoughtRepository`）
- 使用 UPPER_SNAKE_CASE 命名常量（如 `MAX_CONFIDENCE_SCORE`）
- 使用 camelCase 命名私有成员，前缀为下划线（如 `_privateProperty`）

#### 1.1.3 类型与接口
- 接口名使用 PascalCase，前缀为 `I`（如 `IThoughtRepository`）
- 类型别名使用 PascalCase（如 `ConceptMap`）
- 枚举名使用 PascalCase，枚举值使用 UPPER_SNAKE_CASE（如 `RelationType.DEPENDS_ON`）

### 1.2 缩进与格式

- 使用 2 个空格缩进
- 使用 LF 换行符
- 每行不超过 120 个字符
- 大括号使用 K&R 风格（左大括号在同一行）
- 运算符前后加空格
- 逗号后加空格

**示例**：
```typescript
// 好的做法
function updateModel(proposal: CognitiveProposal): UserCognitiveModel {
  const updatedConcepts = this.updateConcepts(proposal.concepts);
  const updatedRelations = this.updateRelations(proposal.relations);
  
  return new UserCognitiveModel(updatedConcepts, updatedRelations);
}

// 避免的做法
function updateModel(proposal:CognitiveProposal):UserCognitiveModel
{
    const updatedConcepts=this.updateConcepts(proposal.concepts);
    const updatedRelations=this.updateRelations(proposal.relations);
    return new UserCognitiveModel(updatedConcepts,updatedRelations);
}
```

### 1.3 注释规范

#### 1.3.1 函数注释
- 使用 JSDoc 注释函数
- 包含 `@param`、`@returns`、`@throws` 等标签
- 描述函数的功能、参数含义和返回值

**示例**：
```typescript
/**
 * 更新认知模型
 * @param proposal AI 生成的认知结构建议
 * @returns 更新后的认知模型
 * @throws {ValidationError} 当 Proposal 无效时抛出
 */
function updateModel(proposal: CognitiveProposal): UserCognitiveModel {
  // 实现
}
```

#### 1.3.2 代码注释
- 只在必要时添加注释
- 注释应该解释 "为什么"，而不是 "是什么"
- 避免冗余注释
- 使用中文注释

### 1.4 代码组织

- 每个文件只包含一个主要类或函数
- 按照以下顺序组织代码：
  1. 导入语句
  2. 类型定义
  3. 常量定义
  4. 类定义
  5. 函数定义
  6. 导出语句

- 使用 `export` 和 `import` 语句，避免使用 `require()`
- 优先使用命名导出，避免使用默认导出

## 2. 错误处理

### 2.1 错误分类

系统使用以下错误分类：

| 错误类型 | 描述 | 处理方式 |
|----------|------|----------|
| DomainError | 业务规则违反 | 返回 400 Bad Request |
| ValidationError | 输入验证失败 | 返回 400 Bad Request |
| NotFoundError | 资源不存在 | 返回 404 Not Found |
| UnauthorizedError | 认证失败 | 返回 401 Unauthorized |
| ForbiddenError | 权限不足 | 返回 403 Forbidden |
| InfrastructureError | 基础设施错误 | 返回 500 Internal Server Error |
| AIError | AI 处理错误 | 返回 500 Internal Server Error |

### 2.2 错误传播

- 使用 `throw` 抛出具体的错误对象，不抛出字符串
- 错误应该在适当的层级被捕获和处理
- 上层模块应该将底层错误包装为更有意义的错误

**示例**：
```typescript
// 好的做法
try {
  await this.llmClient.generateProposal(thought);
} catch (error) {
  throw new AIError("Failed to generate proposal", error);
}

// 避免的做法
try {
  await this.llmClient.generateProposal(thought);
} catch (error) {
  throw "AI 处理失败";
}
```

### 2.3 错误日志

- 所有错误都应该被记录
- 记录错误的类型、消息、堆栈跟踪和上下文信息
- 使用适当的日志级别（如 ERROR 或 WARN）

## 3. 日志规范

### 3.1 日志级别

| 级别 | 描述 | 使用场景 |
|------|------|----------|
| DEBUG | 调试信息 | 开发环境，详细的调试数据 |
| INFO | 一般信息 | 系统正常运行的状态信息 |
| WARN | 警告信息 | 潜在的问题，但不影响系统运行 |
| ERROR | 错误信息 | 系统错误，需要关注和处理 |
| FATAL | 致命错误 | 系统崩溃，需要立即处理 |

### 3.2 日志格式

日志应该包含以下信息：
- 时间戳（ISO 8601 格式）
- 日志级别
- 模块名称
- 日志消息
- 上下文信息（如请求 ID、用户 ID）
- 错误堆栈（如果是错误日志）

**示例**：
```
2023-06-01T12:00:00.123Z INFO [CognitiveParser] Generated proposal with confidence 0.85
2023-06-01T12:01:30.456Z ERROR [LLMClient] Failed to call LLM API: Request timed out
    at LLMClient.callApi (/app/src/infrastructure/ai/LLMClient.ts:45:15)
    at CognitiveParser.generateProposal (/app/src/infrastructure/ai/CognitiveParser.ts:67:20)
```

### 3.3 日志使用原则

- 不要记录敏感信息（如密码、API 密钥）
- 不要过度记录日志，避免性能问题
- 只记录有价值的信息
- 使用结构化日志，便于分析和监控

## 4. 测试策略

### 4.1 测试层次

系统采用三层测试策略：

1. **单元测试**：测试单个函数或类
2. **集成测试**：测试模块间的交互
3. **API 测试**：测试 HTTP API 接口

### 4.2 测试覆盖率

- 单元测试覆盖率目标：80% 以上
- 集成测试覆盖率目标：60% 以上
- API 测试覆盖率目标：100% 核心接口

### 4.3 测试工具

| 测试类型 | 工具 |
|----------|------|
| 单元测试 | Jest |
| 集成测试 | Jest + Supertest |
| API 测试 | Supertest |
| 类型检查 | TypeScript |
| 代码质量 | ESLint |

### 4.4 测试命名规范

- 测试文件与被测试文件同名，后缀为 `.test.ts`（如 `CognitiveModelService.test.ts`）
- 测试函数使用 `test()` 或 `it()`，描述清晰
- 使用 `describe()` 组织相关测试用例

**示例**：
```typescript
describe('CognitiveModelService', () => {
  describe('updateModel', () => {
    test('should update model with valid proposal', () => {
      // 测试逻辑
    });
    
    test('should throw ValidationError with invalid proposal', () => {
      // 测试逻辑
    });
  });
});
```

## 5. 版本控制

### 5.1 Git 分支策略

- `main`：主分支，包含稳定的生产代码
- `develop`：开发分支，包含最新的开发代码
- `feature/*`：功能分支，用于开发新功能
- `fix/*`：修复分支，用于修复 bug
- `release/*`：发布分支，用于准备发布

### 5.2 提交规范

使用 Conventional Commits 规范，提交信息格式：

```
<类型>[可选的作用域]: <描述>

[可选的正文]

[可选的页脚]
```

**类型**：
- `feat`：新功能
- `fix`：bug 修复
- `docs`：文档更新
- `style`：代码格式修改（不影响功能）
- `refactor`：代码重构（不新增功能或修复 bug）
- `test`：测试相关修改
- `chore`：构建过程或辅助工具的修改

**示例**：
```
feat(cognitive-model): 添加概念置信度计算

添加了概念置信度计算功能，用于评估 AI 生成的概念的可靠性。

Closes #123
```

### 5.3 代码审查流程

1. 开发者创建功能分支并开发
2. 开发者提交代码并创建 Pull Request
3. CI/CD 系统运行测试和代码质量检查
4. 其他开发者审查代码
5. 审查通过后，代码合并到 `develop` 分支
6. 定期从 `develop` 分支创建 `release` 分支，准备发布

## 6. 性能优化

### 6.1 性能测试

- 定期进行性能测试
- 关注关键指标：响应时间、吞吐量、内存使用
- 使用性能测试工具（如 Artillery）

### 6.2 优化策略

1. **数据库优化**：
   - 使用适当的索引
   - 避免 N+1 查询
   - 优化 SQL 查询

2. **AI 调用优化**：
   - 缓存 AI 结果
   - 批量处理请求
   - 优化 Prompt 长度

3. **代码优化**：
   - 避免不必要的计算
   - 使用适当的数据结构
   - 异步处理 IO 操作

4. **缓存策略**：
   - 缓存频繁访问的数据
   - 使用适当的缓存过期策略
   - 考虑使用 Redis 等缓存系统

## 7. 安全性

### 7.1 认证与授权

- 使用安全的认证机制（如 API Key、JWT）
- 实现最小权限原则
- 定期轮换密钥和凭证
- 不要在代码中硬编码密钥

### 7.2 数据保护

- 加密敏感数据
- 实现适当的访问控制
- 定期备份数据
- 遵循数据隐私法规

### 7.3 输入验证

- 验证所有用户输入
- 使用参数化查询，避免 SQL 注入
- 对输入进行适当的转义
- 限制输入长度

### 7.4 API 安全

- 实现速率限制
- 使用 HTTPS
- 添加适当的 HTTP 头（如 CORS、CSRF 保护）
- 隐藏敏感信息（如版本号、服务器信息）

## 8. 开发工作流

### 8.1 开发环境设置

1. 克隆代码库
2. 安装依赖：`npm install`
3. 配置环境变量：复制 `.env.example` 为 `.env` 并填写必要的值
4. 启动开发服务器：`npm run dev`

### 8.2 构建与部署

- 构建命令：`npm run build`
- 测试命令：`npm test`
-  lint 命令：`npm run lint`
- 类型检查命令：`npm run typecheck`

### 8.3 CI/CD 流程

1. 代码提交到 Git 仓库
2. CI 系统自动运行测试和代码质量检查
3. 检查通过后，构建镜像
4. 部署到测试环境
5. 手动或自动测试
6. 部署到生产环境

## 11. 代码审查要点

1. **正确性**：代码是否实现了预期功能？
2. **安全性**：代码是否存在安全漏洞？
3. **性能**：代码是否高效？
4. **可维护性**：代码是否易于理解和维护？
5. **可读性**：代码是否清晰可读？
6. **测试覆盖**：代码是否有足够的测试覆盖？
7. **遵循规范**：代码是否遵循了项目规范？
8. **架构一致性**：代码是否符合系统架构？
