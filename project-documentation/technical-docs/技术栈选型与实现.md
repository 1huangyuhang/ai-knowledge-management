# 技术栈选型与实现

## 1. 技术选型总原则

本系统的技术选型遵循以下核心原则：

1. **新手友好**：选择学习曲线平缓的技术，便于单人开发
2. **行业主流**：采用成熟、广泛使用的技术栈
3. **可替换性**：任何组件都能被替换，不绑定特定供应商
4. **性能与简洁**：在保证性能的前提下，优先选择简洁的技术
5. **AI 友好**：支持与 AI 工具良好协作

## 2. 后端核心技术栈

| 维度 | 技术 | 版本 | 选择原因 |
|------|------|------|----------|
| 语言 | TypeScript | - | 强类型、面向对象、AI 友好，便于大型项目维护 |
| 运行时 | Node.js | 18+ | 生态成熟，支持异步编程，适合 IO 密集型应用 |
| 框架 | Fastify | - | 高性能、低开销、API 设计简洁，比 Express 更高效 |
| 构建工具 | ts-node / tsup | - | 新手友好，便于快速开发和调试 |

## 3. AI 技术栈

| 能力 | 技术 | 选择原因 |
|------|------|----------|
| LLM | 云端 LLM API | 无需本地部署，可灵活切换不同提供商 |
| Embedding | text-embedding | 用于生成文本的向量表示，支持语义相似度计算 |
| Prompt | JSON Schema Prompt | 确保 AI 输出结构化数据，便于系统处理 |

### 3.1 AI 技术约束

- **AI 只能输出 Proposal**：AI 不得直接写数据库，所有输出必须经过 Domain Service 验证
- **Prompt 必须版本化**：便于追踪和管理 Prompt 变更
- **AI 属于 Infrastructure**：AI 实现细节被封装，不影响 Domain 层

## 4. 数据与存储技术栈

| 数据类型 | 存储技术 | 用途 | 选择原因 |
|----------|----------|------|----------|
| 原始输入 | SQLite | 存储用户输入的原始思维片段 | 轻量级、文件型数据库，无需额外服务 |
| 认知模型 | Graph + SQLite | 存储概念图谱和关系 | 结合了关系型数据库的稳定性和图数据库的关系表达能力 |
| 语义向量 | Qdrant | 存储文本的向量表示，用于相似度搜索 | 开源、轻量级、支持多种向量索引算法 |

### 4.1 数据分层策略

| 数据类型 | 存储位置 | 说明 |
|----------|----------|------|
| 原始输入 | SQL | 可回放，用于重新生成认知模型 |
| 认知模型 | Graph + SQL | 可演化，支持复杂的概念关系查询 |
| 语义向量 | Vector DB | 用于相似度计算，支持语义搜索 |

## 5. 各层技术实现细节

### 5.1 Domain Layer（纯净层）

**技术要求**：
- 不依赖任何第三方库
- 不依赖数据库
- 不依赖 AI

**文件结构**：
```
src/domain/
├── entities/              # 核心领域对象
│   ├── CognitiveConcept.ts
│   ├── CognitiveRelation.ts
│   └── UserCognitiveModel.ts
├── services/              # 领域服务
│   └── CognitiveModelService.ts
└── value-objects/         # 值对象
```

**技术要点**：
- 使用不可变对象（尽量）
- 所有业务规则写成方法
- 不允许出现 HTTP / SQL / Prompt 相关代码
- 只关注“什么是认知结构”，不关注“如何实现”

### 5.2 Application Layer（用例层）

**职责**：
- 编排流程
- 调用 Domain 层
- 不处理技术细节

**文件结构**：
```
src/application/
├── usecases/              # 用例实现
│   └── IngestThoughtUseCase.ts
├── ports/                 # 接口定义
│   ├── ThoughtRepository.ts
│   └── CognitiveModelRepository.ts
└── dtos/                  # 数据传输对象
```

**技术要点**：
- 依赖倒置：只依赖接口，不依赖具体实现
- 不直接实例化具体类
- 每个用例只处理一个业务流程
- 包含输入输出 DTO、工作流步骤和错误映射

### 5.3 Infrastructure Layer（技术实现层）

**职责**：
- 实现 Application 层定义的接口
- 处理技术细节
- 与外部系统交互

#### 5.3.1 数据存储实现

**文件结构**：
```
src/infrastructure/persistence/
├── sqlite/                # SQLite 实现
│   ├── ThoughtRepositoryImpl.ts
│   └── CognitiveModelRepositoryImpl.ts
└── vector/                # 向量数据库实现
    └── EmbeddingRepositoryImpl.ts
```

**技术要点**：
- 使用简单 SQL，不做 ORM 魔法
- Repository 只做 CRUD，不包含业务规则
- 支持事务管理
- 实现接口，便于替换不同存储技术

#### 5.3.2 AI 能力实现

**文件结构**：
```
src/infrastructure/ai/
├── LLMClient.ts           # LLM 客户端
├── PromptTemplates.ts     # Prompt 模板管理
├── CognitiveParser.ts     # 认知解析器
└── EmbeddingService.ts    # 向量生成服务
```

**技术要点**：
- Prompt 必须版本化
- 输出必须校验 JSON Schema
- 失败要可恢复，支持重试机制
- 支持多种 LLM 提供商

### 5.4 Presentation Layer（交互层）

**职责**：
- 处理用户交互
- 接收输入，返回输出
- 参数校验

**文件结构**：
```
src/interfaces/http/
├── routes/                # HTTP 路由
│   ├── thought.ts         # 思维片段相关路由
│   └── model.ts           # 认知模型相关路由
└── server.ts              # 服务器配置
```

**技术要点**：
- HTTP 只做参数校验，不写业务逻辑
- 路由设计遵循 RESTful 原则
- 支持 JSON 格式的请求和响应
- 包含适当的错误处理和日志记录

## 6. 技术替换策略

本系统设计为高度可替换，任何核心组件都可以被替换而不影响系统整体架构。

### 6.1 替换 LLM 提供商

1. 实现新的 LLMClient 类，继承自 LLMClient 接口
2. 在配置文件中切换 LLM 提供商
3. 无需修改其他代码

### 6.2 替换数据库

1. 实现新的 Repository 类，实现对应的接口
2. 在依赖注入容器中注册新的实现
3. 无需修改 Application 层和 Domain 层代码

### 6.3 替换 Web 框架

1. 实现新的 HTTP 路由处理
2. 保持 API 接口不变
3. 无需修改 Application 层和 Domain 层代码

## 8. 部署与运行环境

### 8.1 开发环境

- macOS / Linux / Windows
- Node.js 18+
- npm 或 yarn

### 8.2 生产环境

- 建议使用 Docker 容器化部署
- 支持云服务器或本地服务器
- 内存要求：至少 4GB
- 存储空间：根据数据量而定，建议至少 10GB

## 9. 技术实现约束

1. **不使用微服务**：初期采用单体架构，便于开发和维护
2. **不使用前端复杂框架**：初期只提供 API，不开发复杂前端
3. **不使用魔法式 Agent 框架**：保持代码清晰，便于理解和调试
4. **所有 AI 调用必须有日志**：便于追踪和调试
5. **所有接口必须有文档**：使用 Swagger 或类似工具生成 API 文档

## 10. 未来技术演进方向

1. **微服务化**：当系统规模扩大时，可考虑拆分为微服务
2. **前端框架**：根据需求，可引入 React 或 Vue 等前端框架
3. **实时通信**：引入 WebSocket，支持实时反馈和通知
4. **多模态支持**：支持文本以外的输入类型，如语音、图像等
5. **本地模型支持**：支持本地部署的 LLM 模型，提高隐私性
