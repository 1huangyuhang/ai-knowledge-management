# 代码生成指导文档

## 1. 项目概述

### 1.1 项目目标
AI认知辅助系统旨在持续建模并分析用户认知结构，输出结构反馈与思考方向。系统关注概念主干、层级、断裂与空洞，帮助用户优化认知结构。

### 1.2 核心业务流程
输入 → 解析（LLM Proposal）→ 建模（Domain 规则）→ 反馈 → 建议

### 1.3 架构原则
- **Clean Architecture**：分层架构，清晰的依赖方向
- **DDD（领域驱动设计）**：核心业务逻辑封装在领域层
- **AI能力接口化**：AI能力通过接口提供，领域层不直接依赖AI
- **高内聚、低耦合**：各模块职责清晰，依赖关系简单

## 2. 技术栈与架构

### 2.1 技术栈
- **运行环境**：Node.js LTS（≥18）
- **开发语言**：TypeScript（严格模式）
- **Web框架**：Fastify
- **AI集成**：LLM + Embedding（接口封装）
- **数据库**：SQLite → PostgreSQL → Qdrant
- **测试框架**：Vitest
- **苹果服务集成**：苹果认证服务（Sign in with Apple）、苹果推送通知服务（APNs）

### 2.2 分层架构
```
┌─────────────────┐
│ Presentation   │  # API层、WebSocket服务
├─────────────────┤
│ Application     │  # Use Case、工作流编排
├─────────────────┤
│ Domain          │  # 核心领域模型、业务规则
├─────────────────┤
│ Infrastructure  │  # 数据库、缓存、外部服务
├─────────────────┤
│ AI Capability   │  # AI能力接口封装
└─────────────────┘
```

|------|------|------|

## 4. 代码生成流程

### 4.1 初始化阶段
1. 创建项目结构
2. 配置TypeScript和依赖
3. 初始化测试框架
4. 设置基础配置文件

### 4.2 领域层实现
1. 实现核心领域模型（UserCognitiveModel、CognitiveConcept等）
2. 实现领域服务和业务规则
3. 编写领域层单元测试

### 4.3 应用层实现
1. 实现Use Case（Ingest Thought、Generate Proposal等）
2. 实现工作流编排
3. 编写应用层单元测试

### 4.4 基础设施层实现
1. **数据库组件实现**：
   - 实现数据库连接配置（DatabaseConfig）
   - 实现数据库连接管理（DatabaseConnectionManager）
   - 实现PostgreSQL仓库（PostgreSQLUserCognitiveModelRepository等）
   - 实现苹果后端数据库设计（设备令牌表和苹果认证表）
2. **文件系统组件实现**：
   - 实现文件存储服务（LocalFileStorageService）
   - 实现文件上传和处理逻辑
3. 实现缓存机制
4. 实现事件系统
5. 编写基础设施层集成测试

### 4.5 API层实现
1. 实现Fastify路由
2. 实现API请求处理
3. 实现核心控制器：
   - FileUploadController
   - SpeechToTextController
   - AITaskController
   - AppleAuthController（处理苹果认证请求）
   - APNsController（处理苹果推送通知请求）
   - iOSConfigController（处理iOS客户端配置请求）
4. 实现WebSocket服务
5. 编写API层集成测试

### 4.6 AI能力层实现
1. 实现LLM客户端
2. 实现Embedding服务
3. 实现AI输出验证
4. 编写AI能力层测试

### 4.7 测试层实现
1. 编写单元测试（覆盖核心逻辑）
2. 编写集成测试（覆盖模块间交互）
3. 编写端到端测试（覆盖关键业务流程）

## 6. 代码生成规范

### 6.1 命名规范
- **类名**：PascalCase（如 `UserCognitiveModel`）
- **函数名**：camelCase（如 `ingestThought`）
- **变量名**：camelCase（如 `cognitiveModel`）
- **常量名**：UPPER_SNAKE_CASE（如 `MAX_RETRY_COUNT`）
- **文件名**：kebab-case（如 `user-cognitive-model.ts`）

### 6.2 代码风格
- 使用4个空格缩进
- 每行不超过120个字符
- 适当的空行分隔代码块
- 优先使用类型推导
- 避免深层嵌套（不超过3层）

### 6.3 注释规范
- **文件注释**：每个文件顶部添加版权和功能说明
- **类注释**：说明类的职责和用途
- **函数注释**：说明函数的参数、返回值和副作用
- **复杂逻辑注释**：解释复杂算法或业务规则
- **TODO注释**：标记待完成的工作

### 6.4 错误处理规范
- 使用自定义错误类型（如 `DomainError`、`ApplicationError`）
- 错误信息应包含清晰的原因和上下文
- 避免使用 `throw new Error()` 直接抛出通用错误
- 实现统一的错误处理中间件
- 苹果服务相关错误类型：
  - `AppleAuthError`：苹果认证失败
  - `APNsError`：苹果推送通知失败
  - `AppleTokenError`：苹果令牌验证失败

### 6.5 测试覆盖率要求
- **领域层**：100% 单元测试覆盖率
- **应用层**：100% 单元测试覆盖率
- **基础设施层**：80% 集成测试覆盖率
- **API层**：80% 集成测试覆盖率
- **AI能力层**：70% 测试覆盖率
