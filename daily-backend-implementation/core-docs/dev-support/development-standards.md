## 1. 文档概述

本文档定义了AI认知辅助系统的开发规范，包括代码风格、命名规范、提交规范、架构原则等内容。通过遵循统一的开发规范，可以提高代码质量、可读性和可维护性，促进团队协作，减少开发过程中的冲突和问题。

## 2. 代码风格规范

### 2.1 TypeScript代码风格

- **缩进**：使用2个空格进行缩进
- **行宽**：每行代码不超过100个字符
- **分号**：必须使用分号结尾
- **引号**：使用单引号（`'`），避免使用双引号（`"`）
- **空格**：
  - 运算符前后必须有空格
  - 逗号后必须有空格
  - 冒号后必须有空格
  - 花括号前后必须有空格
- **花括号**：
  - 左花括号与语句在同一行
  - 右花括号独占一行
- **空行**：
  - 函数之间必须有一个空行
  - 代码块之间必须有一个空行
  - 类的成员之间必须有一个空行

### 2.2 变量和常量

- **变量声明**：使用`const`或`let`，禁止使用`var`
- **类型标注**：明确标注变量类型，避免使用`any`
- **初始化**：变量声明时必须初始化
- **常量命名**：使用全大写，单词之间用下划线分隔（`CONSTANT_NAME`）
- **变量命名**：使用小驼峰命名法（`variableName`）

### 2.3 函数和方法

- **函数声明**：使用箭头函数或函数声明
- **参数**：
  - 函数参数必须有类型标注
  - 可选参数放在必需参数之后
  - 使用默认参数值，避免在函数体内进行参数初始化
- **返回值**：明确标注返回值类型
- **命名**：使用小驼峰命名法（`functionName`）
- **简洁性**：函数应保持简洁，单一职责

### 2.4 类和接口

- **类命名**：使用大驼峰命名法（`ClassName`）
- **接口命名**：使用大驼峰命名法，以`I`开头（`IInterfaceName`）
- **继承**：避免深度继承，优先使用组合
- **访问修饰符**：明确使用`public`、`private`或`protected`
- **构造函数**：使用简洁的构造函数语法
- **属性初始化**：在构造函数中初始化属性，或使用属性初始化器

### 2.5 类型定义

- **接口优先**：优先使用接口定义类型，而不是类型别名
- **类型别名**：使用类型别名简化复杂类型定义
- **联合类型**：使用联合类型（`|`）表示多种可能的类型
- **交叉类型**：使用交叉类型（`&`）组合多个类型
- **泛型**：合理使用泛型，提高代码复用性

### 2.6 注释规范

- **文件注释**：每个文件顶部必须包含文件注释，说明文件的功能和用途
- **类注释**：每个类必须包含JSDoc注释，说明类的功能和用途
- **方法注释**：每个公共方法必须包含JSDoc注释，说明方法的功能、参数和返回值
- **复杂代码注释**：复杂的代码块必须包含注释，说明代码的逻辑和意图
- **注释格式**：使用JSDoc格式的注释

## 3. 命名规范

### 3.1 项目结构命名

- **目录命名**：使用小写字母，单词之间用连字符分隔（`directory-name`）
- **文件命名**：使用小写字母，单词之间用连字符分隔（`file-name.ts`）
- **测试文件命名**：在原文件名后添加`.test`或`.spec`（`file-name.test.ts`）
- **配置文件命名**：使用`.env.example`作为配置模板，`.env`作为实际配置文件

### 3.2 变量和常量命名

- **全局变量**：使用全大写，单词之间用下划线分隔（`GLOBAL_VARIABLE`）
- **局部变量**：使用小驼峰命名法（`localVariable`）
- **实例变量**：使用小驼峰命名法（`instanceVariable`）
- **静态变量**：使用小驼峰命名法（`staticVariable`）
- **常量**：使用全大写，单词之间用下划线分隔（`CONSTANT_NAME`）

### 3.3 函数和方法命名

- **函数命名**：使用小驼峰命名法，动词开头（`getUser`, `calculateTotal`）
- **方法命名**：
  - 公共方法：使用小驼峰命名法，动词开头
  - 私有方法：使用小驼峰命名法，下划线开头（`_privateMethod`）
  - getter/setter：使用小驼峰命名法，与属性名一致

### 3.4 类和接口命名

- **类命名**：使用大驼峰命名法，名词或名词短语（`User`, `CognitiveModel`）
- **接口命名**：使用大驼峰命名法，以`I`开头（`IRepository`, `IService`）
- **抽象类命名**：使用大驼峰命名法，以`Abstract`开头（`AbstractRepository`）
- **枚举命名**：使用大驼峰命名法（`EnumName`）
- **枚举成员命名**：使用全大写，单词之间用下划线分隔（`ENUM_MEMBER`）

### 3.5 类型和类型别名命名

- **类型命名**：使用大驼峰命名法（`TypeName`）
- **类型别名命名**：使用大驼峰命名法（`TypeAliasName`）
- **联合类型命名**：使用大驼峰命名法，以`Or`连接（`StringOrNumber`）
- **交叉类型命名**：使用大驼峰命名法，以`And`连接（`UserAndProfile`）

## 4. 架构设计规范

### 4.1 Clean Architecture原则

- **分层设计**：严格遵循Clean Architecture的5层设计
- **依赖倒置**：高层模块不依赖低层模块，两者都依赖抽象
- **接口隔离**：使用接口定义模块间的通信契约
- **单一职责**：每个类和模块只负责一个功能
- **开闭原则**：对扩展开放，对修改关闭
- **里氏替换**：子类可以替换父类而不影响系统功能

### 4.2 分层设计规范

#### 4.2.1 表示层（Presentation Layer）

- **控制器**：处理HTTP请求和响应，调用应用层服务
- **中间件**：处理跨域、认证、日志等横切关注点
- **路由**：定义API路由，映射到控制器
- **DTO**：定义请求和响应的数据传输对象

#### 4.2.2 应用层（Application Layer）

- **用例**：实现业务流程，协调领域层和基础设施层
- **服务**：实现跨领域的业务逻辑
- **工作流**：协调多个用例的执行

#### 4.2.3 领域层（Domain Layer）

- **实体**：定义核心业务实体，包含业务规则
- **值对象**：定义无标识的领域对象
- **领域服务**：实现跨实体的业务逻辑
- **仓库接口**：定义数据访问契约

#### 4.2.4 基础设施层（Infrastructure Layer）

- **数据库**：实现数据库访问，包括ORM、查询等
- **文件存储**：处理文件上传和存储
- **缓存**：实现缓存机制
- **日志**：实现日志记录
- **外部服务**：集成外部API和服务

#### 4.2.5 AI能力层（AI Capability Layer）

- **LLM集成**：实现与大语言模型的集成
- **嵌入服务**：实现文本嵌入生成
- **认知分析**：实现认知模型分析算法

### 4.3 依赖管理规范

- **依赖注入**：使用tsyringe实现依赖注入
- **接口优先**：依赖接口，不依赖具体实现
- **松耦合**：模块间通过接口通信，减少直接依赖
- **可测试性**：依赖可替换，便于单元测试

### 5.2 文件组织

- **单文件单职责**：每个文件只包含一个类、接口或函数
- **相关文件**：将相关的文件放在同一个目录下
- **命名一致性**：文件名与文件内容保持一致
- **模块化**：将相关的功能组织成模块

### 5.3 导入和导出

- **导入顺序**：
  1. 内置模块
  2. 第三方模块
  3. 内部模块
- **导入路径**：使用相对路径，避免使用绝对路径
- **导出**：
  - 使用命名导出，避免使用默认导出
  - 导出声明与定义分离
- **重命名**：避免导入冲突，使用重命名

## 6. 提交规范

### 6.1 提交消息格式

遵循[Conventional Commits](https://www.conventionalcommits.org/)规范，格式如下：

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### 6.2 提交类型

| 类型 | 描述 |
|------|------|
| `feat` | 新增功能 |
| `fix` | 修复bug |
| `docs` | 文档更新 |
| `style` | 代码风格调整（不影响功能） |
| `refactor` | 代码重构（不新增功能，不修复bug） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `build` | 构建配置变更 |
| `ci` | CI/CD配置变更 |
| `chore` | 其他变更（不影响代码、测试、文档等） |
| `revert` | 回滚之前的提交 |

### 6.3 提交范围

- 可选，用于指定提交影响的范围
- 例如：`feat(user): add login endpoint`

### 6.4 提交描述

- 简短、清晰地描述提交内容
- 使用命令式语气（"add", "fix"而不是"added", "fixed"）
- 首字母小写，结尾不加句号

### 6.5 提交正文

- 可选，详细描述提交内容
- 换行符分隔，每行不超过100个字符
- 解释变更的原因和影响

### 6.6 提交脚注

- 可选，包含 breaking changes 和 issues 引用
- **Breaking Changes**：以`BREAKING CHANGE:`开头，描述破坏性变更
- **Issues**：引用相关的issue，例如`Closes #123`

### 6.7 示例

```
feat(user): add login endpoint

Implement POST /api/v1/auth/login endpoint for user authentication

Closes #456
```

```
fix(database): resolve connection leak

Fix connection leak in PostgreSQL repository by properly closing connections

BREAKING CHANGE: Connection pool configuration has changed, please update your .env file
```

## 7. 测试规范

### 7.1 测试覆盖

- **单元测试**：覆盖核心领域模型和业务逻辑，覆盖率≥80%
- **集成测试**：覆盖模块间的集成，覆盖率≥70%
- **系统测试**：覆盖完整的业务流程
- **性能测试**：测试系统的性能和可扩展性
- **安全测试**：测试系统的安全性和漏洞

### 7.2 测试组织

- **测试文件**：与被测试文件放在同一目录下，使用`.test.ts`或`.spec.ts`后缀
- **测试套件**：使用`describe`组织测试套件
- **测试用例**：使用`it`或`test`定义测试用例
- **断言**：使用Jest的断言库

### 7.3 测试命名

- 测试套件：使用大驼峰命名法，与被测试对象一致
- 测试用例：使用描述性名称，说明测试内容
- 例如：
  ```typescript
describe('UserService', () => {
  it('should create a new user', () => {
    // 测试逻辑
  });
});
  ```

### 7.4 测试原则

- **隔离性**：测试用例之间相互独立
- **可重复性**：测试结果应该是可重复的
- **快速执行**：测试应该快速执行
- **清晰性**：测试代码应该清晰易懂
- **维护性**：测试代码应该易于维护

## 8. 错误处理规范

### 8.1 错误类型

- **业务错误**：使用自定义错误类，包含错误码和错误消息
- **系统错误**：使用内置错误类或自定义错误类
- **验证错误**：使用Zod等验证库的错误

### 8.2 错误处理

- **try-catch**：在适当的层级捕获错误
- **错误传递**：将错误传递给上层调用者
- **错误日志**：记录错误信息，包括错误码、错误消息、堆栈跟踪等
- **错误响应**：向客户端返回统一格式的错误响应

### 8.3 错误响应格式

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "ERROR_CODE",
    "message": "Error message",
    "details": []
  },
  "code": 400,
  "message": "Bad Request"
}
```

### 8.4 错误码规范

- **格式**：使用大写字母，单词之间用下划线分隔（`ERROR_CODE`）
- **唯一性**：每个错误码必须唯一
- **分类**：按照功能模块或错误类型分类
- **文档化**：错误码必须在API文档中进行说明

## 9. 安全规范

### 9.1 输入验证

- **所有输入**：对所有用户输入进行验证
- **验证库**：使用Zod进行数据验证
- **验证规则**：明确、严格的验证规则
- **错误反馈**：向用户返回清晰的验证错误信息

### 9.2 认证和授权

- **JWT**：使用JWT进行认证
- **密码安全**：使用bcrypt或Argon2进行密码哈希
- **权限检查**：实现基于角色的访问控制（RBAC）
- **令牌刷新**：实现令牌刷新机制

### 9.3 数据安全

- **数据加密**：敏感数据在传输和存储过程中进行加密
- **数据脱敏**：日志和响应中对敏感数据进行脱敏
- **SQL注入防护**：使用参数化查询或ORM，避免SQL注入
- **XSS防护**：对输出进行转义，避免XSS攻击

### 9.4 安全监控

- **审计日志**：记录重要的安全事件
- **异常检测**：检测和处理异常请求
- **速率限制**：实现API速率限制，防止暴力攻击

## 10. 性能规范

### 10.1 代码性能
