## 1. 概述

本文档详细描述了AI认知辅助系统的多环境构建与管理方案，旨在实现开发、测试和生产三个环境的隔离与独立管理，确保各环境间职能不互相影响。

## 2. 架构设计

### 2.1 环境分层

| 环境类型 | 主要职能 | 特点 |
|---------|---------|------|
| 开发环境 | 日常开发、功能验证 | 配置灵活、便于调试、数据隔离 |
| 测试环境 | 功能测试、性能测试、集成测试 | 配置接近生产、数据独立、自动化测试 |
| 生产环境 | 线上服务、用户访问 | 高可用、高性能、安全可靠、配置严格 |

### 2.2 核心原则

- **环境隔离**：各环境拥有独立的配置、数据库和资源
- **配置管理**：集中化配置，支持环境特定覆盖
- **自动化构建**：统一构建流程，支持环境特定构建选项
- **统一代码库**：使用分支策略管理不同环境的代码

## 3. 配置管理方案

### 3.1 配置分层结构

采用分层配置策略，默认配置与环境特定配置分离，环境特定配置自动覆盖默认配置。详细的配置管理方案请参考 [配置管理](config-management.md) 文档：

```
config/
├── default.json          # 默认配置，所有环境共享
├── development.json      # 开发环境配置，覆盖默认配置
├── test.json             # 测试环境配置，覆盖默认配置
├── production.json       # 生产环境配置，覆盖默认配置
└── custom-environment-variables.json  # 环境变量映射
```

### 3.2 配置内容

| 配置类别 | 核心配置项 |
|---------|-----------|
| 应用配置 | 应用名称、版本、环境标识 |
| 服务器配置 | 端口、主机、超时时间 |
| 数据库配置 | 数据库类型、连接字符串、池大小 |
| 日志配置 | 日志级别、格式、输出方式 |
| AI配置 | API密钥、模型参数、请求超时 |
| 安全配置 | JWT密钥、CORS设置 |

### 3.3 环境变量管理

- 使用`.env`文件管理敏感配置
- 生产环境通过环境变量注入配置
- 配置加载优先级：环境变量 > 环境特定配置 > 默认配置

## 4. 构建与运行脚本

### 4.1 脚本设计

| 脚本名称 | 功能描述 | 环境 |
|---------|---------|------|
| `dev` | 开发模式运行，支持热重载 | 开发环境 |
| `build:dev` | 开发环境构建，不压缩代码 | 开发环境 |
| `build:test` | 测试环境构建，压缩代码 | 测试环境 |
| `build:prod` | 生产环境构建，压缩代码 | 生产环境 |
| `start:dev` | 启动开发环境服务 | 开发环境 |
| `start:test` | 启动测试环境服务 | 测试环境 |
| `start:prod` | 启动生产环境服务 | 生产环境 |
| `test` | 运行测试用例 | 测试环境 |
| `test:coverage` | 运行测试并生成覆盖率报告 | 测试环境 |

### 4.2 构建流程

1. 清理旧构建产物
2. 编译TypeScript代码
3. 生成环境特定配置
4. 打包依赖
5. 生成构建产物

## 5. 环境隔离措施

### 5.1 数据库隔离

- 开发环境：本地SQLite数据库
- 测试环境：独立的测试数据库实例
- 生产环境：高性能数据库集群

### 5.2 资源隔离

- 各环境使用独立的端口
- 日志文件按环境分离
- 缓存服务按环境隔离

### 5.3 网络隔离

- 生产环境配置严格的网络访问控制
- 测试环境与生产环境网络隔离
- 开发环境可访问测试环境，但不可直接访问生产环境

## 6. 分支管理策略

采用Git Flow分支管理策略：

| 分支类型 | 用途 | 命名规则 |
|---------|------|---------|
| 主分支 | 生产环境代码 | `main` |
| 开发分支 | 开发环境代码 | `develop` |
| 特性分支 | 新功能开发 | `feature/feature-name` |
| 发布分支 | 预发布测试 | `release/version` |
| 修复分支 | 生产环境bug修复 | `hotfix/bug-description` |

## 7. CI/CD流程

### 7.1 持续集成

- 开发分支：提交代码自动运行单元测试和代码检查
- 特性分支：合并前自动运行集成测试
- 发布分支：自动构建测试环境，运行完整测试套件

### 7.2 持续部署

- 测试环境：通过CI自动部署
- 生产环境：手动触发部署，需要代码审查和测试通过

### 7.3 部署流程

1. 代码提交到相应分支
2. CI自动运行测试和构建
3. 测试通过后生成部署包
4. 部署到目标环境
5. 运行健康检查
6. 部署成功后通知相关人员

## 8. 监控与日志

### 8.1 日志管理

- 各环境日志独立存储
- 日志级别按环境配置：开发环境（debug）、测试环境（info）、生产环境（warn/error）
- 生产环境日志集中收集和分析

### 8.2 监控指标

- 服务可用性
- 响应时间
- 错误率
- 资源使用率

## 9. 安全措施

### 9.1 配置安全

- 敏感配置通过环境变量注入
- 配置文件中不存储明文密码和密钥
- 生产环境配置加密存储

### 9.2 部署安全

- 生产环境使用HTTPS
- 严格的访问控制和认证机制
- 定期安全扫描和漏洞修复

## 10. 实施步骤

### 10.1 准备工作

1. 安装必要的依赖包
2. 创建配置文件结构
3. 配置环境变量
4. 设置Git分支策略

### 10.2 开发环境配置

1. 配置开发环境数据库
2. 设置开发环境API密钥
3. 配置开发环境日志级别

### 10.3 测试环境配置

1. 配置测试环境数据库
2. 设置测试环境API密钥
3. 配置自动化测试脚本

### 10.4 生产环境配置

1. 配置生产环境数据库
2. 设置生产环境API密钥（通过环境变量）
3. 配置生产环境安全设置
4. 设置监控和告警

## 11. 最佳实践

1. **配置即代码**：将配置文件纳入版本控制（敏感信息除外）
2. **环境一致性**：尽量保持各环境配置一致，只修改必要的环境特定配置
3. **自动化测试**：确保各环境都有相应的测试用例
4. **文档化**：详细记录各环境的配置和部署流程
5. **定期备份**：定期备份生产环境数据
6. **灰度发布**：生产环境更新采用灰度发布策略

## 12. 故障排查

### 12.1 常见问题

- 配置加载错误：检查环境变量和配置文件格式
- 数据库连接失败：检查数据库配置和网络连接
- 服务启动失败：查看日志文件，检查端口占用情况

### 12.2 排查流程

1. 查看服务日志
2. 检查配置文件
3. 检查环境变量
4. 检查依赖版本
5. 检查网络连接
6. 运行健康检查脚本

## 13. 总结

本方案通过分层配置管理、自动化构建脚本、环境隔离措施和严格的分支管理策略，实现了开发、测试和生产三个环境的有效隔离和独立管理。该方案具有以下优势：

- 提高开发效率：开发环境配置灵活，便于调试
- 确保测试质量：测试环境接近生产，测试结果可靠
- 保障生产稳定：生产环境配置严格，安全可靠
- 支持快速部署：自动化构建和部署流程
- 便于维护和扩展：清晰的架构设计和文档

通过实施本方案，可以确保AI认知辅助系统在不同环境下稳定运行，各环境间职能不互相影响，同时支持快速迭代和持续部署。
