## 1. 文档概述

本文档详细描述了认知辅助系统的日志管理策略，包括日志分类、日志级别、日志格式、日志存储、日志查询和日志分析等内容。通过合理的日志管理，可以提高系统的可观测性，便于问题排查、性能优化和安全审计。

## 2. 日志设计原则

### 2.1 核心原则

- **可观测性**：日志应提供足够的信息，便于理解系统运行状态
- **结构化**：使用结构化日志格式，便于日志分析和查询
- **分级管理**：根据日志重要性进行分级，便于过滤和处理
- **高性能**：日志记录不应显著影响系统性能
- **安全性**：日志中不应包含敏感信息，如密码、token等
- **可扩展性**：支持多种日志存储和分析方案

### 2.2 设计目标

1. **全面覆盖**：记录系统各个层级的关键操作和事件
2. **便于排查**：提供足够的上下文信息，便于定位问题
3. **性能优化**：支持异步日志记录，减少对主业务的影响
4. **安全合规**：符合数据保护和隐私法规要求
5. **可分析性**：支持日志聚合和分析，便于发现系统瓶颈和异常

## 3. 日志分类

### 3.1 按功能模块分类

| 模块名称 | 描述 | 主要日志内容 |
|----------|------|--------------|
| **API请求日志** | 记录所有API请求和响应 | 请求方法、路径、状态码、响应时间、请求IP等 |
| **业务操作日志** | 记录核心业务操作 | 认知模型更新、概念添加、关系建立等 |
| **AI任务日志** | 记录AI任务的执行过程 | 任务创建、开始、完成、失败等状态变化 |
| **系统事件日志** | 记录系统级事件 | 服务启动、关闭、配置变更、异常等 |
| **安全审计日志** | 记录安全相关事件 | 登录、授权、访问控制等 |
| **性能监控日志** | 记录系统性能指标 | 响应时间、资源利用率、数据库查询时间等 |

### 3.2 按日志级别分类

| 级别 | 描述 | 使用场景 |
|------|------|----------|
| **FATAL** | 致命错误，导致系统无法继续运行 | 数据库连接失败、核心服务崩溃等 |
| **ERROR** | 错误事件，可能导致功能异常 | API调用失败、业务逻辑错误等 |
| **WARN** | 警告事件，需要关注但不影响系统运行 | 配置项缺失、资源即将耗尽等 |
| **INFO** | 信息事件，记录系统正常运行状态 | 服务启动、API请求成功等 |
| **DEBUG** | 调试信息，用于开发和测试阶段 | 变量值、函数调用栈等 |
| **TRACE** | 跟踪信息，用于详细调试 | 每一步执行流程、参数传递等 |

## 4. 日志格式

### 4.1 结构化日志格式

系统使用JSON格式的结构化日志，便于日志分析和查询：

```json
{
  "level": 30, // 日志级别数值
  "time": 1704624000000, // 时间戳（毫秒）
  "pid": 12345, // 进程ID
  "hostname": "server-01", // 主机名
  "service": "cognitive-assistant", // 服务名称
  "version": "1.0.0", // 服务版本
  "module": "api", // 模块名称
  "requestId": "req-123456789", // 请求ID
  "userId": "user-123456789", // 用户ID
  "event": "api_request", // 事件类型
  "method": "POST", // HTTP方法
  "url": "/api/v1/input/text", // 请求URL
  "status": 200, // 响应状态码
  "responseTime": 150, // 响应时间（毫秒）
  "ip": "192.168.1.1", // 请求IP
  "message": "API request completed successfully" // 日志消息
}
```

### 4.2 日志字段说明

| 字段名 | 类型 | 必选 | 描述 |
|--------|------|------|------|
| `level` | number | 是 | 日志级别数值（10: TRACE, 20: DEBUG, 30: INFO, 40: WARN, 50: ERROR, 60: FATAL） |
| `time` | number | 是 | 时间戳（毫秒） |
| `pid` | number | 是 | 进程ID |
| `hostname` | string | 是 | 主机名 |
| `service` | string | 是 | 服务名称 |
| `version` | string | 是 | 服务版本 |
| `module` | string | 是 | 模块名称 |
| `requestId` | string | 否 | 请求ID（用于关联同一请求的所有日志） |
| `userId` | string | 否 | 用户ID（如果适用） |
| `event` | string | 是 | 事件类型 |
| `message` | string | 是 | 日志消息 |
| 其他字段 | any | 否 | 业务相关的自定义字段 |

## 5. 日志存储

### 5.1 本地日志存储

- **存储位置**：`/var/log/cognitive-assistant/`
- **文件命名格式**：`service-name-date.log`（如：`api-2024-01-08.log`）
- **日志滚动策略**：
  - 按大小滚动：单个日志文件最大100MB
  - 按时间滚动：每天生成新的日志文件
  - 保留期限：本地日志保留7天

### 5.2 集中式日志存储

系统使用ELK Stack（Elasticsearch + Logstash + Kibana）进行集中式日志管理：

- **Elasticsearch**：用于日志存储和索引
- **Logstash**：用于日志收集和处理
- **Kibana**：用于日志查询和可视化

### 5.3 日志传输

- 使用Filebeat采集本地日志文件
- 日志通过TLS加密传输到Logstash
- 支持批量传输，提高性能
- 支持断点续传，确保日志不丢失

## 6. 日志配置

### 6.1 配置文件结构

```yaml
logging:
  level: info # 全局日志级别
  format: json # 日志格式：json或text
  timestamp: iso # 时间戳格式：iso或epoch
  service: cognitive-assistant # 服务名称
  version: 1.0.0 # 服务版本
  
  # 本地日志配置
  file:
    enabled: true
    path: /var/log/cognitive-assistant/
    maxSize: 100MB # 单个文件最大大小
    maxFiles: 7 # 保留文件数量
    
  # 集中式日志配置
  elk:
    enabled: true
    host: logstash.example.com
    port: 5044
    tls: true
    
  # 模块级日志配置
  modules:
    api:
      level: debug
    ai:
      level: info
    database:
      level: warn
```

### 6.2 环境变量配置

可以通过环境变量覆盖配置文件中的设置：

| 环境变量 | 描述 | 默认值 |
|----------|------|--------|
| `LOG_LEVEL` | 全局日志级别 | `info` |
| `LOG_FORMAT` | 日志格式 | `json` |
| `LOG_FILE_ENABLED` | 是否启用本地文件日志 | `true` |
| `LOG_ELK_ENABLED` | 是否启用ELK日志 | `false` |
| `LOG_ELK_HOST` | Logstash主机 | `localhost` |
| `LOG_ELK_PORT` | Logstash端口 | `5044` |

## 7. 日志使用指南

### 7.1 日志记录最佳实践

1. **包含足够上下文**：每条日志应包含足够的信息，便于理解事件发生的背景
2. **避免敏感信息**：日志中不应包含密码、token、个人隐私数据等
3. **使用结构化字段**：尽量使用结构化字段，避免在消息中拼接信息
4. **统一事件命名**：使用统一的事件命名规范，便于日志分析
5. **合理设置日志级别**：根据事件重要性选择合适的日志级别
6. **关联请求ID**：同一请求的所有日志应包含相同的requestId，便于跟踪

### 7.2 日志查询示例

使用Kibana查询日志：

1. **查询特定用户的所有日志**：
   ```
   userId: "user-123456789"
   ```

2. **查询API请求失败的日志**：
   ```
   module: "api" AND status >= 400
   ```

3. **查询响应时间超过500ms的请求**：
   ```
   module: "api" AND responseTime > 500
   ```

4. **查询特定时间段的错误日志**：
   ```
   level: ERROR AND @timestamp > "2024-01-08T00:00:00Z" AND @timestamp < "2024-01-09T00:00:00Z"
   ```

### 7.3 日志分析和告警

1. **日志分析**：
   - 使用Kibana创建日志仪表盘，实时监控系统运行状态
   - 定期分析日志，发现系统瓶颈和异常
   - 使用机器学习算法检测异常日志模式

2. **日志告警**：
   - 配置基于日志的告警规则，如：
     - 连续5分钟内ERROR日志超过10条
     - 响应时间超过1秒的请求占比超过5%
     - 登录失败次数超过5次
   - 告警方式：邮件、Slack、短信等
   - 告警级别：根据事件严重性设置不同的告警级别

## 8. 日志管理流程

### 8.1 日志收集流程

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   应用服务      │     │   Filebeat      │     │   Logstash      │
├─────────────────┤     ├─────────────────┤     ├─────────────────┤
│  生成日志文件   │────▶│  采集日志文件   │────▶│  处理日志数据   │
└─────────────────┘     └─────────────────┘     └─────────────────┘
                                                      │
                                                      ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   告警系统      │     │   Kibana        │     │   Elasticsearch │
├─────────────────┤     ├─────────────────┤     ├─────────────────┤
│  发送告警通知   │◀────│  日志查询可视化  │◀────│  存储日志数据   │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

### 8.2 日志保留策略

| 日志类型 | 保留期限 | 存储位置 |
|----------|----------|----------|
| 本地日志文件 | 7天 | 服务器本地磁盘 |
| Elasticsearch索引 | 30天 | Elasticsearch集群 |
| 归档日志 | 1年 | 对象存储（如S3） |
| 审计日志 | 7年 | 安全存储 |

### 8.3 日志清理流程

1. **本地日志清理**：使用logrotate工具自动清理过期日志
2. **Elasticsearch索引管理**：
   - 定期创建新索引（如按天）
   - 自动删除过期索引
   - 对旧索引进行压缩和归档
3. **归档日志管理**：
   - 将过期日志导出到对象存储
   - 实现冷热数据分离
   - 提供归档日志查询接口

## 9. 日志安全

### 9.1 访问控制

- 日志文件权限设置为644，仅允许所有者写入
- Elasticsearch访问需要身份认证和授权
- Kibana访问需要登录认证
- 日志查询权限基于角色控制

### 9.2 数据加密

- 日志传输过程中使用TLS加密
- 敏感日志数据在存储前进行脱敏处理
- 归档日志使用AES-256加密存储

### 9.3 审计日志

- 记录所有日志访问操作
- 审计日志独立存储，不可篡改
- 定期审查审计日志，发现异常访问

## 10. 日志监控和优化

### 10.1 日志监控指标

- **日志量**：每分钟生成的日志条数
- **错误率**：ERROR级别日志占总日志的比例
- **日志延迟**：日志从生成到存储的延迟时间
- **存储利用率**：日志存储占用的磁盘空间

### 10.2 日志性能优化

- 使用异步日志记录，减少对主业务的影响
- 合理设置日志级别，避免过多的DEBUG和TRACE日志
- 对大日志进行分片处理
- 实现日志采样，减少日志量
- 优化日志格式，只包含必要的字段

### 10.3 日志质量评估

定期评估日志质量，包括：
- 日志覆盖率：是否覆盖了所有关键业务流程
- 日志准确性：日志信息是否准确反映了实际情况
- 日志可读性：日志是否易于理解和查询
- 日志性能：日志记录是否影响系统性能

## 11. 实施步骤

### 11.1 阶段1：基础配置

1. **安装日志组件**：
   - 安装Elasticsearch、Logstash、Kibana
   - 安装Filebeat

2. **配置日志格式**：
   - 统一应用日志格式为JSON
   - 配置必要的日志字段

3. **配置本地日志存储**：
   - 设置日志文件路径和滚动策略
   - 配置logrotate

### 11.2 阶段2：集中式日志管理

1. **配置Filebeat**：
   - 配置Filebeat采集本地日志
   - 配置TLS加密传输

2. **配置Logstash**：
   - 配置Logstash接收Filebeat数据
   - 配置日志过滤和转换规则
   - 配置输出到Elasticsearch

3. **配置Elasticsearch**：
   - 创建索引模板
   - 配置索引生命周期管理
   - 配置副本和分片

### 11.3 阶段3：日志查询和告警

1. **配置Kibana**：
   - 创建日志索引模式
   - 创建日志仪表盘
   - 配置日志查询

2. **配置告警规则**：
   - 配置基于日志的告警规则
   - 配置告警通知方式
   - 测试告警功能

3. **优化日志配置**：
   - 根据实际使用情况调整日志级别
   - 优化日志字段和格式
   - 调整日志保留策略
