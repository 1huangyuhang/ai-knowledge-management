# 分层存储架构设计

## 1. 架构概述

分层存储架构是AI认知辅助系统数据库实施的重要演进方向，旨在将业务数据库和分析数据库分离，优化不同类型数据的存储和访问方式。本设计文档详细描述了分层存储架构的设计原则、组件构成、数据流向和实施策略。

## 2. 设计原则

### 2.1 核心原则

- **分离关注点**：业务数据库专注于OLTP（在线事务处理），分析数据库专注于OLAP（在线分析处理）
- **优化性能**：针对不同的工作负载优化存储和查询引擎
- **可扩展性**：支持业务数据和分析数据的独立扩展
- **数据一致性**：确保业务数据和分析数据之间的最终一致性
- **成本效益**：根据数据访问频率和重要性选择合适的存储介质

### 2.2 技术选型原则

- **业务数据库**：保持使用PostgreSQL 14，适合OLTP工作负载
- **分析数据库**：选择ClickHouse，适合高吞吐分析查询
- **数据同步**：选择Debezium + Kafka，支持实时数据变更捕获和传输
- **数据格式**：使用Avro或Parquet进行数据序列化，提高存储效率和查询性能

## 4. 数据模型设计

### 4.1 业务数据库模型

业务数据库保持现有的关系型数据模型，专注于支持实时业务操作：

- **核心实体**：UserCognitiveModel、CognitiveConcept、CognitiveRelation、ThoughtFragment、CognitiveInsight
- **设计原则**：
  - 范式化设计，确保数据一致性
  - 适当的索引优化，提高查询性能
  - 支持事务操作

### 4.2 分析数据库模型

分析数据库采用列式存储和星型/雪花模型，优化分析查询性能：

- **核心模型**：
  - 事实表：cognitive_events、concept_interactions、insight_generations
  - 维度表：dim_users、dim_concepts、dim_time、dim_interaction_types
- **设计原则**：
  - 反范式化设计，减少表连接
  - 列式存储，提高分析查询性能
  - 分区和排序键优化

## 5. 数据流向设计

### 5.1 实时数据同步流程

1. **数据变更捕获**：Debezium监控PostgreSQL的WAL日志，捕获数据变更
2. **事件发布**：Debezium将数据变更转换为事件，发布到Kafka主题
3. **事件处理**：Kafka Connect或Flink消费Kafka事件，进行数据转换和清洗
4. **数据加载**：处理后的数据加载到ClickHouse分析数据库
5. **索引更新**：ClickHouse自动更新相关索引

### 5.2 批量数据同步流程

对于历史数据或大规模数据迁移，采用批量同步方式：

1. **数据导出**：使用PostgreSQL的COPY命令或pg_dump导出数据
2. **数据转换**：使用ETL工具（如Airflow）进行数据转换
3. **数据加载**：使用ClickHouse的INSERT SELECT或COPY命令加载数据
4. **验证**：验证数据完整性和一致性

## 6. 存储策略

### 6.1 数据生命周期管理

| 数据类型 | 存储位置 | 保留期限 | 访问频率 | 存储介质 |
|---------|---------|---------|---------|---------|
| 实时业务数据 | PostgreSQL | 永久 | 高频 | SSD |
| 历史业务数据 | PostgreSQL（分区表） | 12个月 | 中频 | HDD |
| 分析数据 | ClickHouse | 永久 | 低频 | HDD/对象存储 |
| 热缓存数据 | Redis | 7天 | 极高 | 内存 |

### 6.2 分区策略

- **PostgreSQL**：按时间分区，每月创建新分区
- **ClickHouse**：按时间和业务维度分区，支持多级分区
- **Kafka**：按主题分区，支持基于键的分区策略

## 7. 性能优化

### 7.1 查询优化

- **业务数据库**：
  - 优化索引设计
  - 使用Prepared Statements
  - 合理设置连接池参数

- **分析数据库**：
  - 选择合适的排序和分区键
  - 使用物化视图加速常用查询
  - 优化JOIN操作，减少数据扫描

### 7.2 数据加载优化

- **实时同步**：
  - 批量加载数据，减少写入次数
  - 调整Kafka分区和消费者组配置
  - 优化Debezium捕获频率

- **批量加载**：
  - 使用高效的数据格式（Parquet/Avro）
  - 并行加载数据
  - 关闭或延迟索引更新

## 8. 数据一致性保障

### 8.1 一致性模型

- **业务数据库**：强一致性，支持ACID事务
- **分析数据库**：最终一致性，数据同步延迟<5分钟
- **缓存层**：最终一致性，通过事件驱动失效

### 8.2 一致性保障机制

- **事务日志**：依赖PostgreSQL的WAL日志确保数据变更的完整性
- **消息确认**：Kafka的ACK机制确保消息不丢失
- **幂等性设计**：数据加载过程支持幂等操作，避免重复数据
- **数据验证**：定期验证业务数据库和分析数据库的数据一致性

## 15. 结论

分层存储架构设计将业务数据库和分析数据库分离，针对不同的工作负载优化存储和查询引擎，提高系统的可扩展性和性能。通过Debezium + Kafka + ClickHouse的技术栈，实现了实时数据同步和高效分析查询。

本设计遵循了分离关注点、优化性能、可扩展性和成本效益的原则，能够支持AI认知辅助系统的长期发展需求。实施过程中需要注意数据一致性、系统复杂度和成本控制等风险，通过合理的实施计划和监控机制确保项目成功。
