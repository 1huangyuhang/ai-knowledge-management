# 统一代码生成指南

## 文档说明

### 文档目的
本统一代码生成指南整合了AI认知辅助系统前端和后端的所有代码生成相关文档，旨在为AI提供一个完整的参考，指导其高效生成高质量代码，解决生成过程中的上下文中断问题，并确保生成的代码符合整体架构设计。

### 文档结构
1. **项目总览** - 项目背景、目标和技术栈
2. **完整文件结构** - 前后端统一的文件结构设计
3. **开发策略** - 开发顺序和策略分析
4. **代码生成框架** - 代码生成流程和规范
5. **模块索引系统** - 所有模块的关联索引和依赖关系
6. **进度跟踪机制** - 进度跟踪和断点恢复
7. **测试框架** - 测试生成和执行策略
8. **AI辅助功能** - 提高生成效率和准确性的策略
9. **文档分区策略** - 基于AI上下文窗口的文档管理

### 文档使用说明
1. AI应按照文档顺序阅读和执行代码生成任务
2. 每次生成前检查模块的生成状态，从上次中断处继续
3. 生成完成后更新模块状态和生成日志
4. 生成过程中遇到问题可参考相关文档
5. 按照文档中的测试策略生成和执行测试

## 1. 项目总览

### 1.1 项目背景和目标
AI认知辅助系统是一个持续构建、更新、分析用户认知结构模型，并基于该模型给出结构性反馈与思考建议的系统。系统采用Clean Architecture设计理念，将核心业务逻辑与外部依赖分离，确保系统的可维护性和可扩展性。

### 1.2 技术栈

#### 后端技术栈
- **核心框架**：Node.js LTS（≥18）、TypeScript（严格模式）、Fastify、Zod
- **数据库**：SQLite（阶段1）→ PostgreSQL（阶段3）
- **AI能力**：LLM + Embedding（接口封装）
- **认证**：JWT + Apple Authentication
- **测试**：Jest或Vitest

#### 前端技术栈
- **核心框架**：Swift 5.9+、SwiftUI、Combine
- **网络请求**：Alamofire
- **数据持久化**：Core Data
- **语音交互**：Speech Framework
- **测试**：XCTest、CombineTestExtensions

### 1.3 核心功能模块

| 功能模块 | 后端实现 | 前端实现 |
|---------|---------|---------|
| **认证模块** | 用户注册、登录、Token管理、Apple认证 | 登录页面、注册页面、认证状态管理 |
| **认知模型模块** | 创建、查询、更新、删除认知模型 | 认知模型列表、详情、创建编辑 |
| **思想片段模块** | 收集、分析用户思想片段 | 思想片段列表、创建 |
| **AI对话模块** | 与AI进行自然语言对话 | AI对话UI、对话功能 |
| **认知分析模块** | 分析用户认知结构 | 多维度分析UI、分析功能 |
| **洞察生成模块** | 生成认知洞察和建议 | 认知洞察展示 |
| **可视化模块** | 生成认知模型可视化数据 | 认知模型可视化、可视化编辑 |
| **语音交互模块** | - | 语音识别、文本转语音 |
| **WebSocket模块** | 实时通信 | WebSocket连接、事件处理 |

### 1.4 前后端关系
- **API设计**：前后端通过RESTful API通信，使用JWT认证
- **数据模型**：前后端共享相同的核心数据模型
- **架构对齐**：均采用Clean Architecture原则，分层设计
- **开发同步**：前后端开发应保持同步，确保API兼容性

## 3. 开发策略

### 3.1 开发顺序分析

#### 策略1：后端优先
- **优点**：确保API稳定后再开发前端，减少前后端对接问题
- **缺点**：前端开发需要等待后端API完成，可能延长整体开发时间
- **适用场景**：API设计复杂，需要后端先行定义数据模型

#### 策略2：前端优先
- **优点**：前端可以快速迭代UI，验证用户体验
- **缺点**：后端需要适配前端设计，可能导致频繁API变更
- **适用场景**：UI/UX设计是核心，需要快速验证

#### 策略3：并行开发
- **优点**：前后端同时开发，缩短整体开发时间
- **缺点**：需要良好的API设计规范和沟通机制
- **适用场景**：有明确的API规范，团队协作良好

### 3.2 推荐开发策略

基于AI认知辅助系统的特点，推荐采用**并行开发策略**，但需要遵循以下原则：

1. **先定义核心数据模型**：前后端团队共同定义核心数据模型和API规范
2. **分层开发**：按Clean Architecture分层开发，先实现领域层，再实现应用层和基础设施层
3. **模块化开发**：按功能模块并行开发，确保模块间低耦合
4. **API优先设计**：先设计API文档，再实现前后端
5. **持续集成**：定期集成前后端代码，及时发现和解决问题

### 3.3 开发优先级

1. **核心数据模型**：前后端共同定义
2. **认证模块**：确保系统安全
3. **认知模型模块**：核心功能
4. **API服务**：前后端通信基础
5. **AI集成**：系统核心AI功能
6. **UI组件**：前端基础组件
7. **语音交互**：增强用户体验
8. **WebSocket**：实时通信
9. **测试优化**：确保代码质量

## 4. 代码生成框架

### 4.1 代码生成流程

#### 4.1.1 准备阶段
1. **检查依赖**：确认当前模块的所有依赖已生成完成
2. **阅读相关文档**：仔细阅读当前模块的技术实现文档和相关核心文档
3. **检查模块状态**：查看进度跟踪文件，确认模块的当前状态
4. **准备生成环境**：确保开发环境配置正确

#### 4.1.2 生成阶段
1. **生成代码文件**：根据技术实现文档生成代码文件
2. **遵循代码规范**：严格按照项目的代码风格指南生成代码
3. **实现所有功能点**：确保实现技术文档中描述的所有功能
4. **添加必要注释**：为核心逻辑和复杂代码添加注释
5. **处理错误情况**：实现适当的错误处理和边界情况处理

#### 4.1.3 验证阶段
1. **检查代码完整性**：确保所有功能点已实现
2. **运行编译检查**：确保代码可以成功编译
3. **运行静态分析**：检查代码是否符合静态分析规则
4. **生成测试代码**：根据模块功能生成相应的测试代码
5. **运行测试**：确保测试通过

### 4.2 代码风格规范

#### 4.2.1 TypeScript（后端）
- 遵循TypeScript严格模式
- 使用ES模块语法
- 遵循Clean Architecture设计原则
- 采用依赖注入模式
- 类名：大驼峰命名法（如`UserService`）
- 方法名：小驼峰命名法（如`createUser`）
- 属性名：小驼峰命名法（如`accessToken`）
- 枚举名：大驼峰命名法（如`UserRole`）
- 常量名：大写蛇形命名法（如`MAX_RETRY_COUNT`）
- 文件名：小写蛇形命名法（如`user.service.ts`）

#### 4.2.2 Swift（前端）
- 遵循Swift API设计指南
- 使用SwiftUI进行UI开发
- 采用Combine进行异步编程
- 遵循MVVM架构
- 类名：大驼峰命名法（如`APIService`）
- 方法名：小驼峰命名法（如`request`）
- 属性名：小驼峰命名法（如`accessToken`）
- 枚举名：大驼峰命名法（如`APIEndpoint`）
- 常量名：大写蛇形命名法（如`MAX_RETRY_COUNT`）

### 4.3 注释规范
- 为类和方法添加文档注释
- 为复杂逻辑添加单行注释
- 为关键算法添加详细注释
- 使用Markdown格式编写文档注释

## 6. 进度跟踪机制

### 6.1 进度跟踪文件

- **文件名**：`CODE_GENERATION_PROGRESS.md`
- **格式**：Markdown
- **位置**：每个项目根目录

### 6.2 跟踪文件内容

```markdown
# 代码生成进度跟踪

## 生成状态总览
- **总模块数**：XX
- **已完成模块数**：XX
- **进行中模块数**：XX
- **未开始模块数**：XX

## 模块生成状态

### 阶段X：XXX
| 模块ID | 模块名称 | 生成状态 | 开始时间 | 完成时间 | 生成日志 |
|--------|----------|----------|----------|----------|----------|
| XXX | XXX | 未开始 | - | - | - |

## 生成日志

## 断点恢复信息
- **最后生成模块**：XXX
- **最后生成时间**：XXX
- **最后生成文件**：XXX
- **恢复建议**：XXX
```

### 6.3 生成状态定义

- **未开始**：模块尚未开始生成
- **进行中**：模块正在生成中
- **已完成**：模块生成完成
- **生成失败**：模块生成失败，需要重新生成

### 6.4 生成日志格式

```
[生成时间] [模块ID] [模块名称] [生成状态] [详细信息]
```

- **生成时间**：YYYY-MM-DD HH:MM:SS
- **模块ID**：如P1W1M1
- **模块名称**：如"项目初始化"
- **生成状态**：开始生成、生成中、生成完成、生成失败
- **详细信息**：如生成的文件路径、遇到的问题等

### 6.5 断点恢复机制

#### 断点恢复步骤
1. **检查进度跟踪文件**：查看最后生成的模块和状态
2. **检查生成的文件**：确认已生成的文件和内容
3. **确定恢复点**：根据最后生成的文件和模块状态确定恢复点
4. **恢复生成**：从恢复点继续生成剩余代码
5. **更新进度**：恢复生成后更新进度跟踪文件

#### 断点恢复示例

```
# 生成中断情况
[2026-01-10 10:30:00] P1W1M2 领域对象定义 生成中 正在生成UserCognitiveModel类

# 恢复步骤
1. 检查进度跟踪文件，发现P1W1M2模块处于"生成中"状态
2. 检查已生成的文件，发现UserCognitiveModel类已部分生成
3. 确定恢复点：从UserCognitiveModel类的剩余部分开始生成
4. 恢复生成UserCognitiveModel类的剩余部分
5. 继续生成其他相关文件
6. 完成后更新模块状态为"已完成"
```

## 7. 测试框架

### 7.1 测试类型和覆盖范围

#### 7.1.1 后端测试类型
- **单元测试**：测试单个函数或类的功能
- **集成测试**：测试模块之间的交互
- **API测试**：测试API端点的功能
- **性能测试**：测试系统的性能表现

#### 7.1.2 前端测试类型
- **单元测试**：测试单个组件或函数的功能
- **集成测试**：测试组件之间的交互
- **UI测试**：测试用户界面的功能和交互
- **性能测试**：测试应用的性能表现

#### 7.1.3 测试覆盖范围
- **Domain层**：纯单元测试（无AI/DB）
- **Application层**：Mock Infrastructure
- **Infrastructure层**：集成测试
- **API层**：API测试
- **核心逻辑**：必须有测试覆盖

### 7.2 测试生成步骤

#### 7.2.1 单元测试生成步骤
1. **确定测试目标**：根据模块功能确定需要测试的类和函数
2. **创建测试文件**：在项目的测试目录下创建测试文件
3. **编写测试用例**：
   - 测试正常功能
   - 测试边界情况
   - 测试错误处理
4. **运行测试**：确保测试通过
5. **更新测试**：当模块功能变更时更新测试

#### 7.2.2 集成测试生成步骤
1. **确定集成点**：确定模块之间的集成点
2. **创建集成测试文件**：在项目的测试目录下创建集成测试文件
3. **编写集成测试用例**：测试模块之间的交互
4. **运行集成测试**：确保集成测试通过
5. **更新集成测试**：当模块交互变更时更新集成测试

### 7.3 测试框架和工具

#### 7.3.1 后端测试框架

- **内容**：进度跟踪机制、测试框架、AI辅助功能
