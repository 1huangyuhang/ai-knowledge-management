主技术规范文档（AI-Executable Master Specification）

文档定位（极其重要）

本文档不是“给人看的技术说明”，而是：
给 AI 作为【总技术宪章 + 可执行规范】使用的文档。

目标效果：
	•	任意高级 AI 在完整阅读本文档后
	•	能自行拆分子文档、子模块、子任务
	•	并在不违背整体架构的前提下
	•	完整实现一个中大型、可维护、可扩展的 AI 软件系统

⸻

1. 项目技术级总体定义（不可更改）

1.1 项目技术性质
	•	项目规模：中大型单体系统（Modular Monolith）
	•	架构范式：Clean Architecture + Domain-Driven Design（DDD）
	•	AI 定位：系统级外部能力（System Dependency），非系统核心
	•	数据形态：
	•	结构化数据
	•	图结构数据
	•	向量语义数据

❗ 本项目不是 Demo、不是 PoC、不是实验性脚本。

⸻

1.2 技术完成态（Definition of Technical Done）

当以下条件全部满足时，项目在技术层面被视为“完成”：
	1.	系统可持续运行，不依赖人工干预
	2.	任意 AI Provider 可被替换而不影响 Domain
	3.	Domain 层可脱离 AI 单独运行
	4.	所有关键模块均有明确接口契约（Contract）
	5.	项目文档可被另一 AI 直接接管并继续开发

3. Domain 层技术完全规范（最高优先级）

3.1 Domain 的唯一职责

Domain 层负责回答以下问题：
	•	什么是“认知结构”？
	•	什么是“概念”？
	•	什么样的关系是合法的？
	•	认知模型在时间维度上如何演化？

Domain 层不关心：
	•	AI 是否存在
	•	数据如何存储
	•	用户如何交互

⸻

3.2 Domain 核心对象定义（强制）

3.2.1 UserCognitiveModel
	•	唯一标识一个用户的认知结构
	•	内部包含：
	•	Concept Graph
	•	Relation Set
	•	Evolution History

3.2.2 CognitiveConcept
	•	表示一个抽象思维单元
	•	属性：
	•	semanticIdentity
	•	abstractionLevel
	•	confidenceScore

3.2.3 CognitiveRelation
	•	表示概念间的逻辑关系
	•	类型示例：
	•	depends_on
	•	generalizes
	•	contradicts

⸻

3.3 Domain Service 强制规范
	•	Domain Service 是唯一允许修改认知模型的入口
	•	所有 AI 输出必须先转为 Proposal
	•	Proposal 必须通过 Domain Service 校验

⸻

4. Application 层技术规范（可拆解）

4.1 Application 的角色
	•	不实现规则
	•	不保存数据
	•	只负责编排流程

⸻

4.2 Use Case 标准结构

每一个 Use Case 必须包含：
	•	Input DTO
	•	Output DTO
	•	Workflow Steps
	•	Error Map

⸻

5. AI Capability 技术协议（极重要）

5.1 AI 输出的唯一合法形式

AI 不允许输出自然语言作为系统输入。

唯一合法输出：

CognitiveProposal {
  concepts: ConceptCandidate[]
  relations: RelationCandidate[]
  reasoningTrace: string[]
  confidence: number
}


⸻

5.2 Prompt Contract
	•	Prompt 必须版本化
	•	Prompt 属于 Infrastructure
	•	Prompt 修改不得影响 Domain

⸻

6. 数据与存储技术规范

6.1 数据分层

数据类型	存储技术	说明
原始输入	SQL	可回放
认知模型	Graph + SQL	可演化
语义向量	Vector DB	相似度


⸻

6.2 Repository 规范
	•	Repository 只做 CRUD

⸻
