# 多环境构建方案设计

## 1. 配置管理优化

只生成相应技术文档就可以了，不要生成代码

### 1.1 配置文件结构

在 `backend` 目录下创建以下配置文件结构：

```
├── config/
│   ├── default.yml          # 默认配置
│   ├── development.yml       # 开发环境配置
│   ├── test.yml              # 测试环境配置
│   └── production.yml        # 生产环境配置
└── .env                      # 本地开发环境变量（gitignore）
```

### 1.2 配置内容示例

**default.yml**：

```yaml
application:
  name: cognitive-api
  version: 1.0.0
  debug: false

server:
  port: 3000
  host: 0.0.0.0
  https: false

logging:
  level: info
  consoleEnabled: true
  fileEnabled: false
```

**development.yml**：

```yaml
application:
  debug: true

logging:
  level: debug
  fileEnabled: false

# 开发环境特定配置
```

**test.yml**：

```yaml
server:
  port: 3001

# 测试环境特定配置
```

**production.yml**：

```yaml
server:
  port: 8080
  https: true

logging:
  level: info
  consoleEnabled: false
  fileEnabled: true
  filePath: /var/log/cognitive-api.log

# 生产环境特定配置
```

## 2. 构建脚本配置

### 2.1 更新 package.json

在 `backend/package.json` 中添加以下脚本：

```json
"scripts": {
  "dev": "NODE_ENV=development tsx watch src/index.ts",
  "build:dev": "NODE_ENV=development tsup src/index.ts --minify false",
  "build:test": "NODE_ENV=test tsup src/index.ts --minify true",
  "build:prod": "NODE_ENV=production tsup src/index.ts --minify true",
  "start:dev": "NODE_ENV=development node dist/index.js",
  "start:test": "NODE_ENV=test node dist/index.js",
  "start:prod": "NODE_ENV=production node dist/index.js",
  "test": "NODE_ENV=test jest",
  "test:coverage": "NODE_ENV=test jest --coverage"
}
```

### 2.2 安装必要依赖

```bash
npm install --save-dev tsx tsup @types/node
npm install --save config dotenv
```

## 3. 代码修改

### 3.1 配置加载逻辑

创建 `src/infrastructure/config/config.loader.ts`：

```typescript
import * as dotenv from 'dotenv';
import * as config from 'config';
import { AppConfig } from './config.types';

// 加载 .env 文件
dotenv.config();

// 加载配置
export const loadConfig = (): AppConfig => {
  return config as AppConfig;
};
```

### 3.2 配置类型定义

创建 `src/infrastructure/config/config.types.ts`：

```typescript
export interface AppConfig {
  application: {
    name: string;
    version: string;
    debug: boolean;
  };
  server: {
    port: number;
    host: string;
    https: boolean;
  };
  logging: {
    level: string;
    consoleEnabled: boolean;
    fileEnabled: boolean;
    filePath?: string;
  };
  // 其他配置项...
}
```

### 3.3 在应用中使用配置

修改 `src/index.ts` 以使用加载的配置：

```typescript
import Fastify from 'fastify';
import { loadConfig } from './infrastructure/config/config.loader';
import { logger } from './infrastructure/logging/Logger';

const config = loadConfig();
const fastify = Fastify();

// 使用配置
fastify.listen({
  port: config.server.port,
  host: config.server.host
}, (err, address) => {
  if (err) {
    logger.error(err);
    process.exit(1);
  }
  logger.info(`Server running on ${address}`);
});
```

## 4. 容器化支持

### 4.1 Dockerfile

创建 `backend/Dockerfile`：

```dockerfile
# 基础镜像
FROM node:20-alpine

# 设置工作目录
WORKDIR /app

# 复制 package.json 和 package-lock.json
COPY package*.json ./

# 安装依赖
RUN npm ci --only=production

# 复制构建后的代码
COPY dist ./dist

# 复制配置文件
COPY config ./config

# 暴露端口
EXPOSE 3000

# 启动命令
CMD ["node", "dist/index.js"]
```

### 4.2 Docker Compose 配置

创建 `docker-compose.yml`：

```yaml
version: '3.8'

services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=${NODE_ENV:-development}
    volumes:
      - ./config:/app/config
      - ./.env:/app/.env
    depends_on:
      - db
  
  db:
    image: postgres:15-alpine
    environment:
      - POSTGRES_DB=${DB_NAME}
      - POSTGRES_USER=${DB_USER}
      - POSTGRES_PASSWORD=${DB_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

## 5. 分支管理策略

### 5.1 建议使用 Git Flow

* **main**：生产环境代码

* **develop**：开发环境代码

* **feature/**：新功能开发分支

* **release/**：发布分支

* **hotfix/**：紧急修复分支

### 5.2 分支流程

1. 从 `develop` 分支创建 `feature/` 分支开发新功能
2. 功能完成后合并回 `develop` 分支
3. 从 `develop` 分支创建 `release/` 分支准备发布
4. 测试通过后合并到 `main` 和 `develop` 分支
5. 生产环境问题从 `main` 分支创建 `hotfix/` 分支修复
6. 修复完成后合并到 `main` 和 `develop` 分支

## 6. CI/CD 配置

### 6.1 GitHub Actions 示例

创建 `.github/workflows/ci-cd.yml`：

```yaml
name: CI/CD Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main, develop ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Use Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'
      - run: npm ci
      - run: npm test

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Use Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run build:${{ github.ref == 'refs/heads/main' && 'prod' || 'dev' }}
      # 构建 Docker 镜像
      - name: Build Docker image
        run: docker build -t cognitive-api:${{ github.sha }} .
      # 推送 Docker 镜像到 registry
      - name: Push Docker image
        run: docker push cognitive-api:${{ github.sha }}

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment: ${{ github.ref == 'refs/heads/main' && 'production' || 'development' }}
    steps:
      # 部署到对应环境
      - name: Deploy to ${{ github.ref == 'refs/heads/main' && 'production' || 'development' }}
        run: |
          # 部署脚本
          echo "Deploying to ${{ github.ref == 'refs/heads/main' && 'production' || 'development' }} environment"
```

## 7. 环境隔离保障

### 7.1 配置隔离

* 不同环境使用独立的配置文件

* 敏感信息通过环境变量注入

* 生产环境配置使用加密存储

### 7.2 数据库隔离

* 开发、测试、生产使用独立的数据库实例

* 测试环境使用测试数据，定期清理

* 生产环境数据定期备份

### 7.3 资源隔离

* 不同环境使用独立的服务器或容器

* 生产环境使用更高规格的资源

* 测试环境使用自动化测试，不影响开发和生产

## 8. 实施步骤

1. **配置管理**：创建配置文件结构和加载逻辑
2. **构建脚本**：更新 package.json 添加多环境构建脚本
3. **代码修改**：修改应用代码使用配置系统
4. **容器化**：创建 Dockerfile 和 Docker Compose 配置
5. **分支管理**：实施 Git Flow 分支策略
6. **CI/CD**：配置 GitHub Actions 自动化构建和部署
7. **测试验证**：在各环境测试构建和部署流程
8. **文档更新**：更新开发文档，记录多环境构建流程

## 9. 预期效果

* 开发、测试、生产环境完全隔离，互不影响

* 支持一键构建和部署不同环境

* 配置管理更加灵活，易于维护

* 支持自动化 CI/CD 流程

* 便于团队协作和版本管理

通过实施这个多环境构建方案，可以确保项目在不同阶段的职能不会互相影响，同时提高开发效率和部署可靠性。
